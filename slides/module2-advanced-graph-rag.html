원리 — RRF `1/(k+rank)` 유도

Module 2 · 3.7
  • 아이디어: 각 검색기에서의 순위(rank)만 신뢰하고 점수는 버린다
  • 한 문서의 RRF 점수 = 각 검색기에서 1/(k+rank)의 합
  • rank=1이면 큰 점수, 순위가 낮아질수록 빠르게 작아짐
  • k의 의미: 상위 순위의 영향력을 완충하는 상수(보통 60)
  • k가 작으면 1등에 과도하게 쏠림 / k가 크면 순위 차이가 평탄
  • 점수 스케일이 달라도 됨 → dense·sparse를 그대로 섞을 수 있음
기억할 한 줄
💡 쉽게 말하면: "1등에게 큰 점수, 등수 내려갈수록 작은 점수"를 두 표에서 각각 주고 더하기 k는 "1등을 너무 우대하지 않게" 눌러 주는 손잡이

RRF의 핵심을 유도로 설명합니다. 왜 점수가 아니라 순위인가? dense의 코사인 점수와 BM25 점수는 스케일이 전혀 달라 직접 더할 수 없기 때문입니다. 순위는 둘 다 1,2,3으로 동일 척도라 더할 수 있습니다. k=60은 관례값이며, k가 작으면 1등 쏠림, 크면 평탄해진다는 직관을 꼭 전하세요. 여러 검색기에 일반화되는 점도 강점입니다.

손으로 계산해 보는 RRF (겁먹지 말기)

k=60으로 점수 계산 (1/(k+등수))
  • 문서 X: 사서 A에서 1등, 사서 B에서 3등
  • 문서 Y: 사서 A에서 2등, 사서 B에는 없음
  • X = 1/(60+1) + 1/(60+3) = 0.0164 + 0.0159 = 0.0323
  • Y = 1/(60+2) = 0.0161
  • X가 더 높음 → 양쪽에서 추천받은 문서가 이긴다

코드 — RRF 융합 함수

def rrf_fuse(dense_ranks, sparse_ranks, k=60):
scores = {}
for ranks in (dense_ranks, sparse_ranks): # 두 랭킹 리스트
for rank, doc_id in enumerate(ranks, start=1):
scores[doc_id] = scores.get(doc_id, 0) + 1.0 / (k + rank)
return sorted(scores, key=scores.get, reverse=True)

정의 — Bi-encoder vs Cross-encoder

bi-encoder: 쿼리·문서를 따로 / cross-encoder: 함께 넣어 비교
Bi-encoder (1차)
  • 인코딩: 쿼리·문서 따로
  • 속도: 빠름(사전 임베딩 가능)
  • 정밀도: 낮음
  • 용도: 대규모 1차 검색
Cross-encoder (재정렬)
  • 인코딩: (쿼리, 문서) 함께
  • 속도: 느림(쌍마다 추론)
  • 정밀도: 높음
  • 용도: 소수 후보 재정렬
Module 2 · 4.2
45
핵심 구조 차이 표입니다. bi-encoder는 문서를 미리 벡터로 만들어 두므로 검색이 빠르지만 쿼리를 볼 때 문서를 다시 보지 못합니다. cross-encoder는 (쿼리,문서) 쌍을 통째로 넣어 토큰 수준 상호작용까지 보지만, 쌍마다 추론해야 해서 전체 코퍼스엔 못 씁니다. 이 트레이드오프가 2단계 구조를 강제한다는 게 다음 슬라이드입니다. 💡 쉽게 말하면: 리랭킹 = 채용 과정. bi-encoder(1차)=수많은 이력서를 빠르게 걸러 30명 추리는 서류전형, cross-encoder(2차)=그 30명을 한 명씩 자세히 보는 면접. 면접을 전 국민에게 할 순 없으니 먼저 서류로 좁히는 것.

왜 그래프인가 — 벡터 RAG의 한계

"벡터는 유사한 청크를, 그래프는 연결된 사실을 검색한다. 둘은 보완재"
  • 벡터 RAG는 청크를 독립적으로 검색 → 청크 간 관계에 약함
  • Multi-hop(멀티홉 — 두 단계 이상 건너뛰며 답 찾기) 실패: "A사가 투자한 회사의 CEO는?" → 두 청크가 따로면 연결 못 함
  • 전역 질문 실패: "보고서 전체의 핵심 리스크는?" → 단일 청크로 답 불가
  • → 엔티티와 관계를 그래프로 명시하면 연결·다단계·전역 질의가 가능

커뮤니티 요약 — 끼리끼리 모인 그룹을 한 줄로 정리하기
  • 커뮤니티 탐지: 밀접하게 연결된 노드들을 군집으로 묶음(Leiden/Louvain — 군집 찾는 알고리즘)

Local vs Global search

질문 유형에 따라 라우팅: 구체·관계형 → Local, 전역·주제형 → Global
Local search
  • 대상: 특정 엔티티 중심
  • 방식: 이웃·관계·청크 수집
  • 질문 예: "X와 Y의 관계는?"
Global search
  • 대상: 전체 / 커뮤니티 요약
  • 방식: map-reduce 요약 종합
  • 질문 예: "전반의 주요 리스크는?"
